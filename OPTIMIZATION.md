# Forge Duel 游戏优化标准

本文件定义了 Forge Duel 游戏的优化标准和目标。遵循这些标准，确保代码达到工业级质量。

---

## 一、客户端性能

### 1.1 渲染优化

| 标准 | 当前状态 | 目标 |
|------|----------|------|
| 60 FPS 稳定帧率 | 未测 | 最低55FPS |
| 减少 draw call | 存在冗余 | 合并批次，目标 <50/帧 |
| 视口外对象销毁 | 无裁剪 | 超出视口 200px 即销毁 |
| 纹理Atlas使用 | 使用单图 | 所有小纹理合并为Atlas |

**具体要求：**
- [ ] 所有游戏内图标（道具、子弹、特效）必须使用 Spritesheet/TextureAtlas 合并
- [ ] 禁用 `autoCull`，手动管理对象池
- [ ] 粒子效果使用对象池复用，禁止 `add.existing` 每帧创建
- [ ] 背景层静态化，使用 `tileSprite` 而非每帧重绘

### 1.2 内存管理

- [ ] 禁止在 `update`/`postUpdate` 中创建任何对象（Vector2、Point等需复用）
- [ ] 所有 `setTimeout`/`setInterval` 必须在对象销毁时同步清除
- [ ] 纹理加载使用缓存检查，禁止重复 `load.image()`
- [ ] 玩家/子弹等对象离开视口后进入休眠池，不销毁

### 1.3 网络同步

| 问题 | 优化方案 |
|------|----------|
| 位置同步冗余 | 只同步增量变化（速度向量），而非全量坐标 |
| 广播风暴 | 使用 Socket.IO Room 隔离，旁观者不收爱好者数据包 |
| 延迟补偿 | 客户端预测 + 服务端校验，误差超过阈值再纠正 |

## 二、服务端性能

### 2.1 架构

- [ ] 关键游戏逻辑（碰撞检测、伤害计算）必须服务喘执行
- [ ] 禁止将玩家输入直接作为伤害源，必须经过服务喘校验
- [ ] 帧率与服务端 tick 率解耦，服务端固定 20tick/s

### 2.2 数据库

- [ ] 玩家数据批量写入缓冲，异步刷盘，禁止每操作写一次
- [ ] 使用 MongoDB 聚合管道替代内存计算
- [ ] 热点数据（天梯排名）使用 Redis 缓存，5分钟过期

---

## 三、代码质量

### 3.1 结构

| 问题 | 标准 |
|------|------|
| 单文件过大 | 单文件不超过 500 行 |
| 泥球代码 | 模块间禁止循环依赖 |
| 魔法数字 | 所有常量定义在 `constants.js`，禁止硬编码 |

**强制规范：**
- [ ] `server.js` 拆分为：`server.js`（入口）+ `routes/` + `game/` + `socket/`
- [ ] 所有游戏配置（伤害值、冷却时间、速度）提取到 `config/game.json`
- [ ] 工具函数归入 `utils/` 目录

### 3.2 Type Safety

- [ ] 所有 Socket 事件 payload 定义接口（interface）
- [ ] 游戏对象（Player、Bullet、Ship）使用 class 或 TypeScript interface
- [ ] 禁止 `any` 类型，特殊情况需注释说明

### 3.3 错误处理

- [ ] 所有 `socket.on()` 必须包裹 `try-catch`
- [ ] Promise 链必须包含 `.catch()` 处理
- [ ] 服务端崩溃需写入错误日志并断开相关连接

---

## 四、可维护性

### 4.1 代码风格

- [ ] 注释使用 JSDoc 规范：函数必须包含 `@param` `@returns` `@description`
- [ ] 变量命名：业务变量用英文，临时变量用 `tmp` / `i` 等通用命名
- [ ] 禁止中文注释（面向国际开源）

### 4.2 测试

- [ ] 核心游戏逻辑（碰撞检测、伤害公式）必须有单元测试
- [ ] Socket 事件必须验证 payload 格式
- [ ] 使用 CI 自动化测试，PR 必须通过测试才能合并

### 4.3 文档

- [ ] `README.md` 包含完整的本地开发指南
- [ ] 每个模块目录包含 `README.md` 说明其职责
- [ ] 架构图使用 Mermaid 绘制

---

## 五、安全性

| 威胁 | 防护措施 |
|------|----------|
| 模拟客户端攻击 | 服务端验证所有输入，客户端数据不可信 |
| 泛洪攻击 | IP 级限流，同一 IP 每秒最多 10 个 Socket 事件 |
| 数据注入 | 所有用户输入转码，MongoDB 使用参数化查询 |
| 权限绕过 | Socket 房间必须验证 `socket.id` 与 `playerId` 匹配 |

## 六、量化指标

优化完成后，项目应达到以下指标：

```
性能:
  - 客户端帧率: ≥55 FPS (独立客户端)
  - 服务端 tick: 稳定 20tick/s, 抖动 <5%
  - 内存占用: 客户端 <200MB (不含浏览器开销)

代码:
  - server.js: <500 行
  - 圈复杂度: <10 (所有函数)
  - 测试覆盖率: 核心逻辑 >80%

安全:
  - 无已知漏洞 (使用 npm audit)
  - 所有输入经过验证
```

## 七、优先级

1. **P0** — 安全漏洞修复
2. **P0** — 崩溃/内存泄漏
3. **P1** — 性能优化（帧率、网络同步）
4. **P1** — 代码结构重组
5. **P2** — 单元测试
6. **P2** — 文档完善

---

## 执行原则

> 每一行代码都应该能回答"为什么存在"这个问题。如果不能，就删掉它。

优化不是重写。保持增量改进，每次 PR 聚焦一个目标。
